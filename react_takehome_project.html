GitHub Repository Explorer

A 2–3 day project to evaluate your ability to build a real-world React application — clean architecture, good component design, and thoughtful performance decisions.

React 18 Hooks Custom hooks GitHub REST API 2–3 days

Overview

Build a GitHub Repository Explorer app that lets users search for a GitHub username and browse that user's public repositories. The app should feel fast, handle edge cases gracefully, and be structured as if it were going into a production codebase.

You will be evaluated on more than just working code — we care deeply about how you structure your components, manage state, handle async complexity, and reason about performance trade-offs.

API

Use the public GitHub REST API — no authentication token required for basic usage (60 requests/hour unauthenticated).

GET
https://api.github.com/users/{`{username}`}
Fetch a user's profile (name, avatar, bio, followers, public repos count)
GET
https://api.github.com/users/{`{username}`}/repos?per_page=30&sort=updated
Fetch the user's public repositories

Required features

User search
  • Search input that accepts a GitHub username
  • Fetches and displays the user profile on submit
  • Shows avatar, name, bio, follower count, and repo count
  • Handles unknown usernames with a clear error message
Repository list
  • Lists all returned repos with name, description, language, and star count
  • Each repo links to its GitHub URL (opens in new tab)
  • Shows a loading skeleton or spinner while fetching
  • Handles empty states (user has no public repos)
Filter & sort
  • Live filter by repo name (client-side, no extra API call)
  • Sort repos by: most stars, most recently updated, or name A–Z
  • Filter by programming language (derive options from the fetched data)
Search history
  • Keep a list of the last 5 searched usernames
  • Clicking a history entry re-fetches that user
  • Persist history in localStorage across page reloads

Technical requirements

These are non-negotiable. Meeting them is the baseline — how well you implement and reason about them is what differentiates candidates.

  • Custom hook — abstract all fetch logic (user + repos) into a custom hook, e.g. useGitHubUser. It should expose loading, error, and data states.
  • Memoization — the filtered and sorted repo list must be derived with useMemo and should not recalculate on unrelated re-renders.
  • Stable callbacks — event handlers passed as props to child components must be stabilised with useCallback where it makes a difference.
  • Abort / cleanup — if the user searches again before a previous request resolves, the stale request must be cancelled or its result ignored.
  • Error boundaries — wrap the main content area in an error boundary so unexpected runtime errors don't crash the entire app.
  • No unnecessary dependencies — do not use a state management library (no Redux, Zustand, Jotai, etc.). Built-in React state and context are sufficient.

What we are not prescribing

The following decisions are yours to make. Be prepared to explain your choices during the review call.

  • Styling approach — CSS modules, Tailwind, styled-components, plain CSS, or anything else
  • Project scaffold — Vite, CRA, or Next.js (but keep it client-rendered)
  • Component structure and file organisation
  • Whether to use TypeScript (encouraged but not required)

Stretch goals

Not scored, but they demonstrate initiative. Only attempt these after the required features are solid.

  • Debounce the search input so it auto-submits after the user stops typing
  • Paginate through repos (the API supports pagination via ?page=)
  • Add a dark/light mode toggle
  • Write at least two unit tests with React Testing Library

Submission instructions

  1. Push your solution to a public GitHub repository.
  2. Include a README.md with setup instructions, your key design decisions, and any trade-offs you made consciously.
  3. The app must run locally with no more than two commands (e.g. npm install & npm run dev).
  4. Reply to this email with the repository link within the agreed deadline.

After submission, we will schedule a 30-minute review call where you will walk us through your code, explain your decisions, and discuss what you would do differently with more time.

Do not spend more than 3 days on this. A focused, well-reasoned solution beats an overengineered one. If something is incomplete, note it in the README rather than leaving broken code.
